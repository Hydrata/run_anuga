## NEEDS_FIX sites

**One NEEDS_FIX (low priority): `check_coordinates_are_in_polygon` at L1325.** The function's shape-sniff `isinstance(coordinates[0], float)` raises `IndexError` if the caller passes `coordinates=[]`. Helpers `_flatten_line_coordinates` and `_extract_polygon_outer_ring` are documented to return `[]` for empty / None geometries. No live caller exercises this path today (every caller short-circuits on empty), so this is a defensive gap rather than a regression. Triaged for the TASK-1106 PR: add a one-line `if not coordinates: return False` guard.

Beyond that single site, every shape-mutable consumer already routes through `_extract_polygon_outer_ring` or `_flatten_line_coordinates` after TASK-1099 (polygon helper) and TASK-1113 (line helper). Audit confirms shape-variance hardening is otherwise complete at the code level; remaining work is documentation (TASK-1106 + this audit), tests (TASK-1108), and the CI gate. The two `# point.AddPoint(...)` references at L661 / L682 are commented-out debug code and were not classified.

## Additional helpers needed?

**Recommendation: no new helper.** The current two-helper set (`_extract_polygon_outer_ring`, `_flatten_line_coordinates`) covers all promotion-class shape variance. Point/MultiPoint has no real promotion class for this codebase:

- Nodes are serialised as scalar `Point` features (single-coord-pair) by Django + GeoServer; MultiPoint is not a normalisation target the BE produces today.
- Even if a MultiPoint slipped through, `shapely.Point(MultiPoint_coords)` raises `TypeError` immediately (fail-fast), unlike the `coordinates[0]` polygon-naive pattern that yielded a silent wrong-shape ring on canary 17.
- The OGR centroid at L599 is always-Point by definition (OGR Centroid() never returns MultiPoint for any input shape).

If TASK-1106 / TASK-1107 later determine that BE serialisation might emit MultiPoint (e.g. multi-station inflow points collapsed into one feature), a `_extract_first_point` helper can be added then. For now this would be speculative.

## Open questions / outstanding decisions

- **TASK-1106 (defensive `or []` defaults decision):** The current helpers return `[]` for missing/empty coordinates, which is a **silent** failure mode. Recommendation: **keep silent + add log warning** at the helper level (DEBUG for None/empty, WARNING for MultiPolygon>1). Rationale: the calling sites then naturally yield empty `interior_regions` / `frictions` / `inflow_dataframe` columns rather than crashing the whole run. A loud `raise` would regress canary stability (one bad feature kills the whole package), whereas silent + WARNING preserves the run while still surfacing the data quality issue in run_anuga logs that already flow back to the control server via HTTPHandler.
- **TASK-1107 (real-multi semantics): SUPERSEDED by the formal decision in the TASK-1107 section below.** The original recommendation here (ship plural helper) was preempted by the orchestrator's Tradeoff-bucket default to the cheaper path (formal REJECT with deferred-design memo). See `## TASK-1107: real-multi support, decision and deferred design` for the current verdict and re-open trigger.
- **TASK-1108 (integration test coverage gaps):** the gaps table above lists 23 missing consumer-site × shape cells. Recommendation: cover at minimum **{Polygon, MultiPolygon} × {empty, None}** for `make_frictions`, `make_interior_regions`, `make_interior_holes_and_tags` (defensive crash protection) and **{LineString, MultiLineString (>1 sub)} for `apply_inflows_to_domain` inflow** (real-multi readiness). Defer the consumer-level multi-warning path (>1 sub-polygon) to TASK-1107 once plural helper lands.
- **`check_coordinates_are_in_polygon` shape sniff at L1325 → NEEDS_FIX (low):** see the NEEDS_FIX sites section above. One-line guard slated for the TASK-1106 PR.

## BE-side canonicalisation interaction (TASK-1116)

If the BE-side canonicalisation in TASK-1116 (sibling task) normalises every geometry to a single canonical shape (Polygon-only, LineString-only) on write, then **every "SAFE" verdict in this audit becomes "trivially safe"**; the helper invocations become defensive no-ops because the input would never carry the Multi variants the helpers were written to handle. The verdicts do not flip in the opposite direction: BE canonicalisation can only reduce the variance the run_anuga side observes, never expand it. The audit's classifications remain valid as a defence-in-depth layer because (a) historic packages serialized before TASK-1116 will still ship Multi-variants to run_anuga, (b) the BE-side canonicalisation runs at write-time so any feature inserted before TASK-1116 lands keeps its existing shape until rewritten, and (c) run_anuga also accepts pre-built packages from external automation that bypass the BE write path. Therefore TASK-1117's audit conclusion ("no NEEDS_FIX sites; current hardening sufficient") is unaffected by TASK-1116's outcome; TASK-1116 is a complementary layer rather than a replacement.

## TASK-1107: real-multi support, decision and deferred design

**Decision: formally REJECT real-multi support for now.** Current behaviour (`_extract_polygon_outer_ring` returns the first sub-polygon and emits a WARNING for >1 sub-polygons) is retained. Reopen this task when canary telemetry warrants it.

### Justification

1. **Empirically working.** Current `warn + take first sub-polygon` behaviour has shipped through canary 17 and TASK-1099 without a single reported case of a dropped sub-polygon affecting a real user run. The audit table above shows every Polygon consumer already routes through the hardened helper.
2. **BE typically emits 1-sub MultiPolygon.** The PostGIS roundtrip artefact in the BE serialiser yields `MultiPolygon` with exactly one sub-polygon for the overwhelming majority of features. A truly-disjoint MultiPolygon with >1 sub-polygon would require a user-drawn feature that the FE map editor does not surface as a common workflow.
3. **Telemetry already exists.** The WARNING at `run_anuga/run_utils.py:546` (`_extract_polygon_outer_ring` helper, message: `"MultiPolygon contains N sub-polygons; using first only"`) flows back to the control server via the HTTPHandler attached to the run_anuga logger. If real-multi geometries hit production we will see it; until then we are speculating.
4. **Cheaper now, reversible later.** The plural helper plus 9 callsite refactor is non-trivial (rainfall/catchment lookup keys, scalar friction tuples) and adds surface area for new shape bugs. The deferred design below makes pickup straightforward if telemetry triggers it.

### Deferred design sketch

Proposed plural helper, to live alongside `_extract_polygon_outer_ring` in `run_anuga/run_utils.py`:

```python
def _extract_all_polygon_outer_rings(geometry: dict | None) -> list[list[tuple[float, float]]]:
    """Return outer rings of every sub-polygon. One list per sub-polygon.

    - Polygon       -> [outer_ring]                    (length 1)
    - MultiPolygon  -> [outer_ring_0, outer_ring_1, ]  (length N)
    - None / empty  -> []
    """
```

### Telemetry trigger to re-open

Re-open TASK-1107 if any of:

- Canary logs show the `MultiPolygon contains N sub-polygons; using first only` warning firing **more than once per week** across the run_anuga deployment, OR
- A user reports a dropped sub-polygon in a friction zone, interior region, interior hole, or catchment, OR
- Product confirms a workflow (e.g. multi-island flood study) that requires authoring disjoint-MultiPolygon features and expects all sub-polygons to take effect.

When re-opened, start with the 5 cheap migrations (interior_regions, interior_holes, both friction sites, area_m2). The 4 non-trivial sites need a product decision on union semantics before code lands.
